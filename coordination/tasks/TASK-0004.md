## Contexto

El dogfood manual del bus `coordination/` (v2 → v3 → v3.1) está **cerrado**. `TASK-0002` quedó en `HUMAN_DECISION` (PR #250) y la `DECISION.md` correspondiente vive en `main`. El siguiente ciclo autorizado es un **Runner Design Review**: una fase de diseño/revisión, **no** de implementación (ver `STATE.md` → "Next cycle" y "Runner status: blocked").

El bus funciona, pero hoy **Jorge sigue siendo cable**: tras cada paso (evidencia lista, review emitido, candidato listo) alguien humano tiene que leer el estado de `main`, decidir qué agente actúa a continuación y abrir las instrucciones. El Runner se concibe para **reducir ese cable humano**, no para sustituir autoridad.

## Objetivo

Diseñar, como **propuesta documental**, un Runner **manual mínimo** que en una futura implementación solo pueda: leer `main`, detectar estados, construir una cola de trabajo, notificar qué agente debe actuar, preparar instrucciones, bloquearse ante ambigüedad y registrar observaciones. El diseño debe dejar explícito qué el Runner **jamás** puede hacer (mergear, aprobar, declarar `human_final`, promover un `DECISION_CANDIDATE`, fijar `HANDOFF_TO_MSO`, tocar MSO/Police/Policy/Auth, ejecutar tareas funcionales, o usar secretos/workflows sin revisión separada).

## Alcance y límites

- **Solo `coordination/`**, y dentro de él solo `tasks/TASK-0004.md` y `proposals/` (`RUNNER_DESIGN_REVIEW.md`, opcionalmente `RUNNER_MANUAL_PROTOCOL_DRAFT.md`).
- Trabajo en rama `coordination/task-0004-runner-design-review`, nunca `main`.
- Entregable normativo: `proposals/RUNNER_DESIGN_REVIEW.md`.

### Non-goals (explícito)

- **No** implementar Runner ni crear scripts ejecutables.
- **No** GitHub Actions, cron, Docker, túnel/VPS, headless automation, tokens/secrets, ni cambios de permisos.
- **No** fijar `HANDOFF_TO_MSO`.
- **No** modificar MSO / Police / Policy / Auth ni contrato/schemas vigentes como implementación definitiva (los cambios viven solo como *draft* en `proposals/`).
- **No** tocar `coordination/STATE.md`, `coordination/tasks/TASK-0002.md`, `coordination/decisions/TASK-0002.DECISION.md`, `coordination/candidates/TASK-0002.DECISION_CANDIDATE.md`, ni `cowork/`.
- **No** cerrar ni mover ninguna tarea existente.

## Criterios de aceptación

- [ ] **AC1 — Problema.** El diseño declara qué problema resuelve el Runner (reducir cable humano) y qué **no** resuelve todavía.
- [ ] **AC2 — Entradas/Salidas.** Tabla de archivos que lee y outputs que produce; el Runner solo **lee** `main` y solo **escribe** observaciones/notificaciones, nunca artefactos de autoridad.
- [ ] **AC3 — Estados y eventos.** Tabla de estados observados (de la máquina v2) y eventos que detecta; para cada uno, qué acción legal propone y a quién notifica.
- [ ] **AC4 — Matriz de permisos.** Matriz explícita de lo que el Runner **puede** y lo que **jamás** puede (mergear, aprobar, `human_final`, promover candidato, `HANDOFF_TO_MSO`, tocar MSO/Police/Policy/Auth, ejecutar, secretos/workflows).
- [ ] **AC5 — Fail-closed.** Reglas fail-closed ante conflicto/ambigüedad: el Runner se bloquea, no asume, no avanza, registra observación.
- [ ] **AC6 — Invariantes V3.1.** Enumera las invariantes de V3.1 que el Runner debe preservar (main autoritativo, authorship != authority, no auto-review, no promoción de candidatos, control de acceso como enforcement).
- [ ] **AC7 — Permisos mínimos futuros.** Permisos mínimos que necesitaría una implementación futura, y por qué deben ser de solo lectura sobre `main` + canal de notificación.
- [ ] **AC8 — Riesgos y pruebas.** Riesgos de seguridad y el dry-run manual necesario antes de cualquier implementación.
- [ ] **AC9 — MVP.** Define el MVP más pequeño posible y el siguiente paso recomendado tras este review.
- [ ] **AC10 — No autoridad.** Deja explícito y no relajable que el Runner es coordinador mecánico conservador, jamás autoridad.
