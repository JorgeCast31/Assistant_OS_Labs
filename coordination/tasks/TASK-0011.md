## Contexto

Assistant OS ya tiene piezas de gobernanza (coordination/ bus v3.1, human approval model, police gate
antes de dispatch en `assistant_os/core/orchestrator.py`, Draft Store, prepared/confirmable actions,
authority trace, Read-Only Queue Reporter mergeado en `scripts/coordination_queue_reporter.py`, y una
Mission Control observacional en `ui/components/sovereign/MissionControlView.tsx`). Lo que falta no son
componentes sino un **plano de coordinación operable** que muestre el ciclo Jorge → MSO → Claude → Codex
→ Jorge sin afirmar ejecución inexistente.

Verdad técnica verificada contra `origin/main` @ `b0fbb5a`:

- Mission Control existe con vistas Planner / MSO / Arms / Orchestration / Outcome.
- `execution_allowed` y `can_execute_now` permanecen invariantemente `False`
  (`assistant_os/mso/authority_binding.py`, `authority_preparation.py`).
- El police gate se aplica antes de dispatch en el orchestrator.
- El Read-Only Queue Reporter es manual, read-only, sin autoridad; Runner permanece bloqueado.
- El endpoint de seat/provider es metadata read-only, no un canal real con Claude/Codex.

## Objetivo

Autorizar **únicamente el diseño documental** de Command Center / Control Plane v0: una propuesta
arquitectónica, una especificación de UI y un backlog de incrementos pequeños y reversibles, todo
**compatible con el contrato vigente de `coordination/`** (sin crear estado paralelo, sin bypass de
autoridad, sin Runner encubierto, sin ejecución ni provider integration).

### Non-goals (explícito, no relajable)

- **No** implementar UI, backend, Runner, provider adapters, cron, polling headless, workflows ni agentes.
- **No** tocar `assistant_os/mso/`, `assistant_os/police/`, `assistant_os/policy/`, auth, secretos,
  `.env`, `.github/workflows/`.
- **No** tocar `coordination/` contratos/schemas, `STATE.md`, `AGENT_CONTRACT.md`,
  `RULES_OF_ENGAGEMENT.md`, `decisions/`, `candidates/`.
- **No** tocar PR #258 ni usarlo como autorización.
- **No** tocar `HANDOFF_TO_MSO`.
- **No** modificar el checkout sucio del usuario, ni `main`, ni mergear.
- **No** presentar provider metadata como integración real con Claude/Codex.

## Criterios de aceptación

- [ ] **AC1.** Existen los tres documentos de diseño con rutas coherentes a la arquitectura.
- [ ] **AC2.** El diseño es una **proyección** compatible con `coordination/TASK.md.status` en `main`; no
  introduce máquina de estados nueva ni store de estado paralelo.
- [ ] **AC3.** Ningún diseño confiere autoridad: `READY`, `HUMAN_DECISION`, `human_final`,
  `HANDOFF_TO_MSO` solo se materializan por eventos verificables de Jorge / MSO. La UI no expone control
  agente-accionable de merge/approve/execute.
- [ ] **AC4.** No hay Runner encubierto ni ejecución: `execution_allowed`/`can_execute_now` permanecen
  `False`; provider = metadata, no canal.
- [ ] **AC5.** El backlog declara que **no autoriza** implementación; cada incremento requiere su propia
  autorización separada (READY en main por Jorge).
- [ ] **AC6.** Solo los seis archivos del scope aparecen en el diff. Áreas prohibidas intactas.
