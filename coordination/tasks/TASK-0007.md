> **BANNER — Propuesta condicional, no autoridad.** Este archivo propone `status: READY` **en una rama**, vía PR. Bajo el contrato v3.1, un agente puede **proponer** una transición de propiedad de Jorge (`READY`) en una rama, pero **no la hace efectiva**. **Proponer-en-rama ≠ fijar.** READY solo será real cuando exista en `main` por **merge verificable de Jorge**. **Si un agente mergea o aprueba este PR, el READY es inválido y nulo** (`RULES_OF_ENGAGEMENT.md §Aclaración v3.1`, `AGENT_CONTRACT.md §Por qué READY también depende del acceso`). `last_legit_status: DRAFT` lo refleja: el último estado alcanzado por transición válida sigue siendo `DRAFT` hasta el merge.

## Contexto

El pipeline Runner/Reporter avanza estrictamente como **diseño → revisión → autorización**, con el Reporter/Runner siempre **bloqueado**:

- `TASK-0004` (Runner Design Review, PR #252) definió el Runner como **coordinador mecánico solo-lectura**, nunca autoridad.
- `TASK-0005` (Runner Manual Dry-Run, PR #253) confirmó **a mano, sin código** que el protocolo es operable sin autoridad, y reveló cuatro fricciones (**F1–F4**).
- `TASK-0006` (Read-Only Queue Reporter Proposal, PR #254) produjo la **especificación documental** del MVP (`proposals/READ_ONLY_QUEUE_REPORTER_MVP.md`) y sus **criterios de prueba** (`proposals/READ_ONLY_QUEUE_REPORTER_ACCEPTANCE_TESTS.md`). Esa spec **no autoriza** nada (su §15 reserva la decisión a Jorge).

El MVP **no está implementado**. El paso natural y mínimo es una **autorización explícita y estrecha** (scope-lock) para que, en un PR posterior, se implemente exactamente el MVP especificado — read-only, sin autoridad — y nada más.

## Objetivo

Producir un **PR documental de autorización y scope-lock** que, **si Jorge lo mergea**, deje autorizada la implementación mínima del `Read-Only Queue Reporter` en un **PR posterior** (tarea nueva). Este PR **no** implementa código.
